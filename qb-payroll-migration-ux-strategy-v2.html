PRD Section 6 — North-star use case · Design every screen against this
"A customer migrates in July and needs to pull an employee's pay stub from February for a housing verification request."

This is the single scenario every UX decision must be tested against. Success = customer navigates to Pay History, clicks February, downloads a stub a mortgage lender will accept — in under 3 minutes, without calling support. Failure = customer still goes to Chart of Accounts, finds nothing, and calls support (an escalation counted against the PRD's primary metric).

Current broken experience
Customer goes to QBO Payroll → finds nothing → navigates to Chart of Accounts → finds a bank transaction → tries to decipher liability and expense rows → frustrated → calls support → escalation counted against primary metric (8/252 = 3.1% of migration issues)
Target experience after this migration
Customer goes to QBO Payroll → Pay History tab → finds employee → clicks February pay period → sees full paycheck detail (earnings, taxes, deductions, net pay) → downloads a cleanly formatted stub → done in under 3 minutes → zero support calls → escalation count = 0
PRD solution decisions — what was considered and why (Section 5)
Discarded — Approach 1
Status quo
Continue migrating as bank transactions and force customers to manually review YTD and gross vs tax. Actively causing escalations and churn. Proven friction point.
Discarded — Approach 2
Pure YTD import only
Import final YTD balances only, no historical paycheck detail. Doesn't solve the need to view individual pay stubs. Creates audit compliance anxiety. Fails the north-star scenario.
Selected — Approach 3
Native historical paycheck creation
Map Desktop paycheck tables directly to QBO/IES HistoricalPaycheck API endpoints. Customer sees a 1:1 reflection of their Desktop payroll history in Pay History. Eliminates manual reconciliation tasks and resolves the lighthouse issue.
PRD success metrics — how we measure whether this worked (Section 7)
Primary metric
8 → 0
IES payroll migration escalations related to historical paychecks. Currently 8 out of 252 total migration issues (3.1%). Every screen in this strategy must be traced to which escalation type it prevents.
Secondary metric
1 → 0
IES customer churn directly attributed to missing historical paycheck functionality. Post-migration, a usage signal monitors whether customers open Pay History within 14 days. Non-openers flagged for proactive outreach.
Guardrail metric
No timeout
Migration must not time out when processing thousands of historical paycheck objects. Period selector limits scope. Chunking strategy required for large files. R15 — open: API rate limit and chunk threshold not confirmed by Engineering.
PRD MECE data taxonomy — what the migration engine must handle (Section 4)
Entity mapping
Accounting entity → Payroll entity
Bank transactions / Chart of Accounts (current) must be replaced by native QBO Paycheck Objects (target). Not supplemented — replaced.
Data granularity
YTD totals + payrun-level detail
Both levels required. YTD for W-2 and tax filing accuracy. Payrun-level (gross pay, specific tax withheld per check) for individual stub review and audit.
Validation stages
System-side + user-facing
System-side: migration engine validates all mappings before posting. User-facing: customer reviews Pay History and has a structured path to report discrepancies — not just a support call.
Three paycheck types in Pay History — all must be visually distinguished (PRD R20)
Live
Live QBO paycheck
Run in QBO after migration. Fully editable. No visual marker needed — this is the default state. These are the records the customer creates going forward.
Historical
Standard historical paycheck
Migrated from Desktop via HistoricalPaycheck API. Read-only — cannot be edited. Badge: "Historical · Migrated from QuickBooks Desktop." Tooltip: "This paycheck was migrated and cannot be edited. Contact support to request a correction."
Override
Read-only historical override
Desktop paycheck with a math error that QBO's tax engine cannot validate. Accepted as-is for historical accuracy. Badge: "Historical · Override." Tooltip: "This paycheck contains a mathematical variance from Desktop that QBO accepted as-is. The figures shown are exactly as they were in Desktop."

AI scanHard gateR01 R15
File health & eligibility scan
AI reads the QBDT file before the customer makes any commitment. File size, version, payroll service type, corruption detection. Go / Caution / Stop signal in plain English.
What happens
AI reads QBDT file: target count vs 750k ceiling, QBDT version, payroll service type (Basic/Enhanced/Assisted), last payroll run date, file corruption indicators. Takes 30–90 seconds. Single progress screen — no forms.
If file exceeds limit: Condense Data option surfaced immediately, before customer invests time in setup. If corrupted: File Doctor repair step shown before anything else.
Why this is required
PRD guardrail metric (R15): migration must not time out. An over-limit file crashes mid-migration with no recovery. Detecting it here saves the customer hours of lost setup work.
Payroll service type determines which QBDT tables are populated. Full Service files have tax payment records that Basic files do not — the migration engine needs this to know what to query.

Business ownerHard gateR16
Migration period selector
Business owner defines which paycheck history to migrate. Defaults to January 1 of the current tax year. Prior-year paychecks are explicitly out of the QBO API scope — directed to the Desktop PDF export fast-follow.
What the screen captures
Period start: defaulted to January 1 current tax year (recommended for full W-2 accuracy). Adjustable — but mid-year start triggers a warning: "Starting mid-year means your YTD totals will only reflect [date] onwards. Your W-2 will show partial-year figures."
Period end: defaulted to last payroll run date from the file scan. Adjustable.
Prior-year toggle: "Do you need paychecks from before January 1 [current year]?" → If yes: "We'll export those as PDFs from Desktop. They won't appear in QBO Pay History but you'll have them as downloadable files." Links to the Desktop PDF export fast-follow.
Live preview: "This will migrate [X] paychecks for [Y] employees. Estimated time: [Z] minutes." Updates as dates change.
Why this is required
PRD requirement R16: explicitly requires "check on the period for which past paychecks are required by user." Without a defined window, the migration engine queries all PAYCHECK_HEADER records, including prior-year — bloating the API payload and risking timeout (guardrail R15).
Scope clarity on prior-year resolves the R21 conflict — PRD's fast-follow (Desktop PDF export) handles prior-year; QBO native migration handles current-year only.

Hard gateAI detectR15
In-progress payroll gate
Detects any payroll currently in progress in QBDT (unsent direct deposit batches, pending paychecks). Hard block — migration cannot begin until this is resolved. Re-scan available once the in-progress payroll is completed.
What the screen shows
If no in-progress payroll detected: green "All clear — no payroll in progress" and customer proceeds to next step.
If detected: "A payroll for [pay period dates] is currently being processed in QuickBooks Desktop. You must complete and send this payroll before migrating. Return here when it's done." Re-scan button available.
Why this is required
QBDT creates PAYCHECK_HEADER records with status=pending before direct deposit files are sent. HistoricalPaycheck API rejects pending-status records. An in-progress payroll mid-migration causes a partial failure with no designed recovery path.
In-progress YTD figures are incomplete — the paycheck is counted but the money hasn't moved. Migrating mid-cycle produces an over-counted YTD that causes the first QBO payroll run to over-withhold taxes.

AI matchingConfirm / overrideR01 R05
Payroll item mapping
Every unique pay item in QBDT mapped to a QBO pay type using AI semantic matching. Progressive disclosure — auto-confirmed items collapsed, ambiguous items reviewed, unmapped items resolved. Negative-value items flagged and re-expressed as deductions.
Three confidence tiers shown
Auto-confirmed (green, collapsed): "32 standard items matched — Federal Income Tax, Social Security, Medicare, Regular Pay, Overtime." One-line summary, no customer action needed.
Review (amber): Medium-confidence — e.g. "Your 'Hlth Ben ER' maps to Employer Health Insurance Contribution. This covers what your company pays toward employees' health insurance — does that look right?" One-click confirm.
Unmatched (red): Customer must select from a searchable dropdown. Plain-English description of each option.
Negative-value detection: "Your 'Expense Reimbursement' item has negative values in Desktop. In QBO, we'll record this as a Deduction item. This keeps your YTD totals accurate." — Explicit one-click confirm.
Why this is required
HistoricalPaycheck API requires every line item to reference a valid QBO pay type ID. One unmapped item blocks the entire paycheck — not just that line. Without this screen, unmapped items silently keep paychecks as bank transactions (the exact problem the PRD is solving).
PRD R05: "full payrun history matching DT records." Employer-side items (FUTA, SUTA, ER FICA match) must also be mapped — they affect Chart of Accounts sync. Open dependency: confirm whether QBO Pay History shows employer-side items or handles them only via CoA.
Open — PRD R02, R05: Confirm whether QBO Pay History stub template includes DT's "company summary" section (employer-side items). If not, document where ER items surface and update the mapping screen to set correct customer expectations.

Customer confirmsComplianceR07 R19
Voided & amended paycheck confirmation
Lists all voided and amended paychecks in the migration window. Customer confirms void reversals are included (required for accurate YTD). Plain-English explanation of the W-2 impact if excluded.
What the screen shows
Count and summary: "We found 3 voided paychecks and 1 amended paycheck within your migration window." Expandable list: employee name, paycheck date, original amount, void date.
Toggle: "Include voided records (recommended)" — ON by default. Plain-English impact if turned off: "Turning this off will overstate your employees' year-to-date wages by [dollar amount]. This will affect their W-2s."
YTD impact preview — shows the difference in YTD totals with vs without voided records included, per affected employee.
Why this is required
A voided paycheck in QBDT has zero net YTD effect — the original paycheck and its void reversal both exist in the data. If only the original migrates (without the void), the employee's YTD is overstated by the full gross amount, producing a W-2 error that requires a W-2c to correct.
This is not an edge case — any company that voided an accidental duplicate paycheck, a paycheck issued to a wrong bank account, or a check that bounced will have voided records.

Pre-flightBlockingR01 R19 R20
Pre-flight validation — dry run before migration commits
Silent API dry-run against every confirmed mapping. Surfaces validation failures, math-error candidates (read-only overrides), and deduplication conflicts. Customer resolves before migration starts — not mid-run.
Four result categories
Ready — passes all payload validations. Will migrate as a native HistoricalPaycheck object.
Read-only override (PRD R20) — math error detected. QBO will accept this paycheck as-is for historical accuracy, but it cannot be edited post-migration. Customer sees: "This paycheck has a small calculation difference from what QuickBooks expected. We'll migrate it exactly as it appears in Desktop, and it will be marked as read-only." Bulk-accept available.
Blocked — API will reject this payload entirely. Three choices: Skip (exclude from migration) / Archive as PDF (preserve as a downloadable file) / Fix mapping (return to mapping screen).
Dedup conflict — paycheck has no matching bank transaction to suppress. Migrating will create a new CoA entry. Customer confirms whether this is expected.
Why this is required
PRD Risk 2: "Historical paychecks in DT might have mathematical inaccuracies that QBO's strict tax engine rejects." Without a dry run, math-error rejections surface mid-migration. At that point the customer has already committed — there is no designed recovery path in the PRD for mid-run failures.
PRD Risk 1: deduplication. Pre-flight checks that each paycheck, if migrated, will correctly suppress its corresponding bank transaction. Any that cannot be deduplicated are flagged before they create doubled CoA entries.

Zero-clickAuto-exportR13
Pre-migration baseline snapshot
System automatically exports Balance Sheet, P&L, Payroll Summary by Employee, and Check Register before migration begins. Stored as reconciliation anchors — the ground truth for post-migration comparison.
What is exported
Balance Sheet (as-of migration date), Profit & Loss (current year YTD), Payroll Summary by Employee (YTD wages, taxes, deductions per employee), Check Register (all payroll bank transactions in the migration window).
Zero customer input — happens automatically. Customer receives email with secure download links. Stored in QBO for 7 years (IRS record retention).
Why this is required
These become the ground truth for the post-migration automated diff (touching PRD R13 — escalation reduction). Without a baseline snapshot, post-migration discrepancies must be reconstructed from memory or paper records, which is what the PRD is trying to eliminate.
If the customer is ever audited post-migration, the baseline snapshot is the evidence that their QBO records are consistent with their QBDT records as of the migration date.

Real-timePause / resumeR01 R15
Migration progress dashboard
Live counters: paychecks migrated / in remediation / archived / exceptions flagged. Real-time ETA. Pause/resume without losing completed work. No black-box spinner — the customer knows exactly where they are.
What is shown
Live counts: "X of Y paychecks migrated as paycheck records" / "Z paychecks archived for compliance" / "N exceptions need your attention." Completion percentage bar.
ETA recalculates based on actual processing rate. If ETA extends beyond a payroll run date, a warning fires: "At this rate, migration will complete after [date] — which is your next scheduled payroll. Do you want to pause and resume after your payroll run?"
Pause/resume: migration state is saved at the last successfully processed paycheck. Customer can close the browser and resume from where they left off.
Why this is required
PRD guardrail R15: migration must not time out. For large files, migration can take hours. Without visibility, customers interrupt or retry — triggering idempotency failures that duplicate paycheck records in QBO.
Pause/resume is a compliance need — a customer must be able to pause before a payroll run date without losing completed work. Without this, they face a choice between interrupting migration (causing data issues) or missing payroll.

AI recoveryCore mechanismR01 R07 R19
Check-to-paycheck intercept
AI catches paychecks the native QBO tool converts to bank transactions, re-matches them to QBDT payroll data, and re-posts them as HistoricalPaycheck objects. Customer sees a recovery count — a success signal, not a warning.
How it works
Match key: paycheck_date + net_pay amount + employee SSN cross-reference + pay_period_end. Multi-field matching prevents false positives.
Matched paychecks: re-enriched with payroll item breakdowns from QBDT data and re-posted as HistoricalPaycheck objects. Customer sees: "14 paychecks recovered from bank transaction format."
Unmatched conversions: surface in the exception queue with the specific check details and three options: Skip / Archive as PDF / Manually link to source paycheck.
Why this is required
This is the core mechanism that resolves the PRD's root cause. The native QBO migration tool converts paychecks to checks by design. Without the intercept layer, every paycheck stays as a bank transaction — Pay History remains empty — and the north-star use case fails.
Bank transaction suppression (PRD R19 deduplication) runs in parallel — the intercepted paycheck triggers suppression of the corresponding journal entry to prevent doubled CoA entries.

AI clusteringCustomer resolvesR13
Exception queue — plain English, three choices
All migration exceptions grouped by root cause. Every exception shows: employee name + paycheck date + specific reason + exactly three resolution options. No error codes. No dead ends. Grouped exceptions resolved in one action.
Exception format
Example: "Sarah Chen · March 15 paycheck · The federal withholding amount (–$42.00) is negative, which QBO does not accept for historical records." → [Skip this paycheck] [Archive as PDF] [Mark as read-only override].
Grouping: if 8 employees all have the same unmapped custom pay item, they appear as one grouped exception with a bulk-resolve action. Never 8 separate alerts for the same root cause.
Categories: Blocking (must resolve before migration completes) / Advisory (can proceed, review recommended) / Informational (logged, no action needed).
Why this is required
PRD primary metric R13: escalations from 8 to 0. Every exception that the customer cannot self-resolve is a potential support call. Plain-English grouping with specific actions keeps resolutions in-product.

Customer sign-offCompliance gateR08 R09
Per-batch YTD validation & sign-off
After each employee batch, side-by-side QBDT vs QBO YTD comparison. Green = match. Amber = explained difference (AI narrates why). Customer signs off per batch. AI explains every delta in plain English — no accountant required for routine rounding differences.
What is compared
Per employee: Gross wages YTD / Federal income tax withheld YTD / Social Security wages and tax / Medicare wages and tax / State income tax per jurisdiction / Each deduction (401k, health insurance, garnishment).
AI explains each amber delta: "The $12.50 difference in John's federal tax is due to QBO using a daily rounding method vs Desktop's per-period method — this is within IRS tolerance and will not affect his W-2."
Customer signs off per batch with a one-click confirmation — "These YTD totals look correct." Incremental review, not one overwhelming final screen.
Why this is required
PRD R08: both YTD aggregate and payrun-level data must migrate correctly. PRD R09: user-facing validation stage required. The first live QBO payroll run reads the migrated YTD accumulator to calculate withholding. An incorrect YTD produces incorrect withholding from the very first QBO paycheck.

Chart of Accounts deduplication & notification
When native paychecks are posted, corresponding bank transactions are suppressed. Customer is proactively shown what changed in their Chart of Accounts and why — preventing the confusion and support call that happens when they see an unexpected change.

Rollback windowDefined recoveryR19
Rollback definition & window
Before migration commits, customer is told exactly what rollback means, what it reverses, what the window is, and how to initiate it. "Can I undo this?" gets a designed answer — not a support call.
What the customer is told
Rollback window: Available until your first live QBO payroll run is submitted. After that, migration changes are permanent.
What rolls back: All historical paycheck records created in QBO, bank transaction restoration in Chart of Accounts, QBO employee records created during migration.
What does not roll back: Your QuickBooks Desktop file is unaffected — it was read-only during migration. Your QBDT file remains as your backup.
How to initiate: Contact support and request a migration rollback. Reference your migration ID: [ID]. The process takes 1–2 business days.
Why this is required
Neither the original strategy document nor the PRD defined this state. "Can I undo this?" is a common post-migration question. Without a designed answer, it becomes a support escalation — the most expensive possible outcome for a decision that takes one screen to pre-empt.

Primary surfaceNorth-star screenR02 R10 R11 R17 R20
Pay History tab — populated with permanent timeline anchor
The screen the PRD's north-star scenario depends on. Historical paychecks visible with permanent timeline anchor. Three paycheck types visually distinguished. Download available. "Report an issue" path for when a stub looks wrong.
Key design decisions
Timeline anchor (permanent): A dated divider in the Pay History list at the migration date, labelled "Migrated from QuickBooks Desktop on [date] — records below this line are historical." This persists forever — not a one-time notification. Six months later, an auditor can identify provenance at a glance.
Three record types: Live QBO (no marker, editable), Historical standard ("Historical" badge, read-only), Read-only override ("Historical · Override" badge, read-only, tooltip explains the math variance).
"Report an issue" path (PRD R09): On each historical paycheck detail view, a "This doesn't match my Desktop records" link. Opens a structured form: "Which amount looks different? What does your Desktop record show?" Routed to the migration support team — not generic support. This is the user-facing validation stage the PRD requires.
Download: Every paycheck (historical and live) has a "Download pay stub" button. Downloaded stub must match DT format for third-party acceptance (open — see dependency note).
Why each element is required
R10: Pay History must be accessible even before the first live QBO payroll run — Engineering must confirm the tab appears when HistoricalPaycheck objects are posted, not only after the first live run.
R11: downloaded stub must match DT format "exactly" for housing verification. The stub format spec (field labels, layout, currency format) must be confirmed as equivalent to DT's template — or a design task to align them is required.
R20: the three-way visual distinction between live / historical / read-only override is essential. Without it, customers try to edit historical records, hit a confusing lock, and call support.
Open — R10, R11: (1) Engineering to confirm Pay History tab appears for migrated-only employees before first live payroll run. (2) Compare DT stub PDF vs QBO stub PDF field-for-field. If layouts differ, design task required to align formats.

Auto-diffReconciliationR13 R09
Automated report comparison
QBO reports diffed line-by-line against pre-migration baseline snapshots. AI categorises every difference as expected, known limitation, or genuine anomaly. Customer only investigates red rows — amber and green are explained automatically.
What is diffed
Balance Sheet vs baseline: payroll tax liability accounts must match. P&L vs baseline: payroll expense accounts by period. Payroll Summary vs baseline: YTD per employee per line item. Check Register: bank transactions suppressed must equal HistoricalPaycheck objects created.
Pre-classified amber items: rounding method differences (QBO daily vs QBDT per-period), FIFO vs average cost inventory changes, audit trail exclusion. Customer sees these with a plain-English explanation, not an unexplained delta.
Why this is required
PRD R09 user-facing validation stage: customer must have a structured way to confirm migration is correct — not just "look at Pay History and hope." This is the structured validation that closes the loop. PRD R13: every unexplained discrepancy is a potential escalation. Automatic categorisation prevents most of them.

Nightly monitor5-day gateR13 R15
Parallel-run monitor & decommission gate
Nightly comparison of QBDT and QBO registers during the parallel period. Customer receives a daily digest. QBDT decommission is only recommended after 5 consecutive clean days — an objective signal, not a subjective judgment call.
Daily digest format
"Everything matched today — Day 3 of 5 clean days." or "2 items need your attention today — [specific records, specific reasons, specific actions]."
Anomaly detection on first 3 live QBO payroll runs: AI compares run totals against QBDT historical patterns. If a run is statistically anomalous (e.g. total net pay 15% lower than rolling average), alert fires before payroll is submitted — not after employees report issues.
Decommission gate: after 5 clean days, system surfaces: "Your migration looks complete and consistent. You can now safely retire QuickBooks Desktop. [How to archive your Desktop file]."
Why this is required
Without a parallel-run monitor, the customer decides subjectively when to retire QBDT. If they decommission too early and a discrepancy surfaces later, there is no reference system. The 5-day gate converts "I feel ready" into an evidence-based decision.
