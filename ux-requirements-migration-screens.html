Employee states — how each state changes what screens are required
Active employee
Currently employed, payroll running in QBO after migration
All 9 EMP_MASTER fields required — missing any one blocks full paycheck migration
Full PAYCHECK_HEADER history migrates as native HistoricalPaycheck objects
YTD from PAYCHECK_TAX and PAYCHECK_LINE_ITEM must exactly match QBO live YTD accumulator at migration date
Pay item mapping must be complete — every PAYCHECK_LINE_ITEM.pay_item_name must map to a QBO pay type before first live payroll run
W-4 data is required for ongoing withholding — not just historical accuracy
Multi-state work_state(s) must be captured if employee worked across state lines
Needs: screens 1–10 (pre), 11–18 (during), 19–26 (post)
Inactive / terminated employee
Terminated — no future payroll, but historical stubs needed for audit & W-2
EMP_MASTER minimum: employee_id, ssn, name, term_date, pay_frequency — address and W-4 not required for read-only history
PAYCHECK_HEADER migrates as structured archive object — not a live HistoricalPaycheck if insufficient EMP_MASTER data
YTD from PAYCHECK_TAX must still be captured — affects W-2 correction eligibility for up to 3 years post-termination
PAYCHECK_LINE_ITEM still required at line-item level — auditors and housing verifications require full stub detail, not just YTD totals
JOURNAL_ENTRY reconciled_flag must be verified — terminated employee paychecks that were never reconciled create CoA discrepancies post-migration
Needs: screens 1, 2, 5, 6, 7 (pre), 11, 13, 15, 17 (during), 20, 22, 24 (post)
Rehired employee
Terminated and rehired within the migration window — two employment periods
Two EMP_MASTER records may exist — system must detect and flag for explicit merge/split decision
Two separate YTD accumulators — merging silently into one produces an incorrect annual W-2
PAYCHECK_HEADER must be partitioned by employment_period_start/end — two separate pay history sequences in QBO
Pay item mapping must be checked per period — rehire may have different pay items, deductions, or benefits than original employment
PAYCHECK_TAX YTD must be split at the termination date — not summed across both periods
Needs: all active screens PLUS screen 4 (rehire detection) and special YTD split handling in screens 13, 14
Critical dependency chain — why order matters
The data tables have a strict FK dependency chain: Employee → Paycheck → Pay Item → Tax → Ledger. A migration screen that tries to map pay items before employee records are confirmed will produce orphaned line items. A screen that posts journal entries before deduplication is complete will double the CoA. Every screen in the pre-migration phase resolves a dependency before the migration engine reads data from it. Screens out of order = data corruption.
FK chain: EMP_MASTER → PAYCHECK_HEADER → PAYCHECK_LINE_ITEM → PAYCHECK_TAX → JOURNAL_ENTRY

S1
Active + Inactive Critical Paycheck table Employee table
File health & migration eligibility scan
Automated scan of the QBDT company file before any user decisions are made. Surfaces file size, version, corruption, payroll service type, and a go/caution/stop signal.
Why this screen exists
The HistoricalPaycheck API has payload limits and the native migration tool has a 750,000-target ceiling. Starting migration on an over-limit file causes a mid-run crash with no graceful recovery path — customer loses hours of setup work
File corruption in QBDT produces mathematically invalid paycheck records that QBO's tax engine rejects. Detecting corruption before setup prevents unexplained failures mid-migration
Payroll service type (Basic, Enhanced, Assisted, Full Service) determines which QBDT paycheck fields are populated. A Full Service file has tax payment records that Basic does not — the migration engine needs this context to know which tables to query
Information captured
Total target count vs 750k ceiling — surfaces Condense Data option if over limit
QBDT version number — determines which paycheck schema version is in use
Last payroll run date — used to pre-populate the migration period end date in screen 3
Payroll service type — determines mandatory vs optional fields in subsequent screens
Corruption flag — triggers file repair step before any other screen is shown
What breaks if skipped
Over-limit file crashes mid-migration with no recovery — all setup work lost
Corrupted paycheck records produce read-only override candidates that look valid until the API rejects them at run-time
Wrong service type assumption causes missing fields in PAYCHECK_TAX queries — tax withholding records either duplicated or absent
Reads from PAYCHECK_HEADER (count) + EMP_MASTER (count, status)

S3
Active + Inactive Critical Paycheck table
Migration period selector
Business Owner (or ProAdvisor acting on behalf) selects the start and end dates of the paycheck history to migrate. Determines scope of every subsequent screen and every API call.
Why this screen exists
PRD Section 8 explicitly requires: "check on the period for which past paychecks are required by user." Without a defined window, the migration engine queries all PAYCHECK_HEADER records — including prior-year paychecks that the PRD defers to the PDF fast-follow. This bloats the API payload and risks timeout (PRD guardrail metric)
The period start date determines the YTD accumulator baseline — if migration starts from January 1 of current year, the full-year YTD is captured. If it starts mid-year, only partial YTD migrates, which will produce an incorrect W-2 if not explicitly handled
For inactive employees, the period must extend to at least their term_date to capture all paychecks in their final YTD accumulator
Information captured
Period start date — defaulted to January 1 of current tax year (recommended); adjustable
Period end date — defaulted to last payroll run date from S1 scan; adjustable
Prior-year scope toggle — "Include prior years?" If yes: prior-year paychecks are queued for PDF export (fast-follow), not API migration
Paycheck count preview — "This will migrate X paychecks for Y employees. Estimated time: Z minutes." Updates when dates change
Mid-period warning — if start date is not January 1, system shows: "Starting mid-year means your YTD totals will only reflect [date] onwards. Is this intentional?"
What breaks if skipped
Unlimited date range → prior-year paychecks included in API payload → timeout on large files → PRD guardrail metric violated
No mid-year warning → partial-year YTD migrated → W-2 shows incorrect annual totals
Inactive employee's paychecks outside the window not captured → W-2 correction eligibility lost
Reads from PAYCHECK_HEADER (paycheck_date, count)
S4
Rehired only Critical — compliance Employee table Paycheck table
Rehired employee — employment period split confirmation
Surfaces every employee with multiple employment periods within the migration window. ProAdvisor confirms: keep as separate periods (separate YTD) or merge (single YTD). W-2 accuracy depends on this decision.
Why this screen exists
QBDT creates a new payroll_schedule_id when an employee is rehired but may reuse the same employee_id. The PAYCHECK_HEADER records for both employment periods share one FK. QBO treats this as one continuous employment history. If not split, the YTD accumulator sums both periods — producing an annual W-2 wage total that is higher than any single employment period earned, which is a W-2 error
The QBO HistoricalPaycheck API requires an employment_period_start for each paycheck. Without a split decision, the migration engine cannot correctly populate this field for rehired employees
Information captured per flagged employee
Period 1: hire_date → term_date, paycheck count, YTD gross wages
Period 2: rehire_date → migration end date (or current term_date), paycheck count, YTD gross wages
ProAdvisor decision: "Keep as two separate periods" (default, recommended) or "Merge into one continuous record" (requires explicit justification note)
If merged: ProAdvisor must confirm the merged YTD total is correct — system shows the sum and flags if it exceeds Social Security wage base (FICA cap risk)
What breaks if skipped
Summed YTD from two periods → W-2 Box 1 overstated → IRS Form W-2c required → employee and employer compliance exposure
FICA over-withheld if merged YTD exceeds Social Security wage base — employee entitled to a refund that shows up as a payroll tax discrepancy
HistoricalPaycheck API call fails if employment_period_start is ambiguous — paycheck migrates as orphaned record
Reads from EMP_MASTER (hire_date, term_date, rehire_date) + PAYCHECK_HEADER (payroll_schedule_id) + PAYCHECK_TAX (ytd_employee per period)
S5
Active + Inactive Critical Pay item table Paycheck table
Payroll item mapping — QBDT pay item → QBO pay type
Every unique pay_item_name in PAYCHECK_LINE_ITEM mapped to a QBO pay type. Progressive disclosure: auto-confirmed / review / unmatched. Negative-value items detected and resolved here.
Why this screen exists
The QBO HistoricalPaycheck API requires every line item in the payload to reference a valid QBO pay type ID. A PAYCHECK_LINE_ITEM with an unmapped pay_item_name causes a payload validation failure for that entire paycheck — not just that line item. One unmapped item blocks the entire paycheck
QBDT allows negative amounts in PAYCHECK_LINE_ITEM (employee repayments, deduction reversals). QBO does not accept negative values in HistoricalPaycheck payload — they must be re-expressed as deduction items. Without this screen, negative values either invert to positive (corrupting YTD) or cause API rejection
employer_portion_flag items in PAYCHECK_LINE_ITEM (FUTA, SUTA, ER FICA, workers comp) must be mapped separately from employee-side items — they write to different QBO accounts and affect the Chart of Accounts sync differently

Why this screen exists
PAYCHECK_TAX stores both employee_amount and employer_amount per tax_type. The QBO HistoricalPaycheck API requires these separated into distinct payload fields. S7
Active + Inactive Watch Paycheck table
Voided & amended paycheck confirmation
Lists all PAYCHECK_HEADER records with status = voided or amended. ProAdvisor confirms: include void reversals (required for accurate YTD) and amended records (include both original and amendment).
Why this screen exists
A voided paycheck in QBDT has a zero net effect on YTD because the original paycheck and its void reversal both appear in PAYCHECK_LINE_ITEM.ytd_amount. If only the original paycheck migrates (status = processed) without the void record (status = voided), the employee's YTD is overstated by the full gross amount of that paycheck
Amended paychecks require both the original and the amended record to migrate — the amendment is a delta, not a replacement. Migrating only the amended version drops the delta from the YTD history, making the audit trail incomplete
This is equally critical for inactive employees — their final YTD must include any voids or amendments processed after their last regular paycheck but before their termination date
Information captured
Count and list of voided paychecks within migration window — employee name, paycheck date, original amount, void date
Count and list of amended paychecks — original amount, amended amount, delta, amendment date
ProAdvisor toggle: "Include voided records" (default: ON) — with plain-English explanation of YTD impact if turned off
YTD impact preview — shows the difference in employee-level YTD totals with and without voided records included
What breaks if skipped
YTD overstated by amount of voided paychecks → W-2 Box 1 incorrect → W-2c required for affected employees
Audit trail incomplete — if an auditor requests the payroll history and the void record is absent, the gross-to-net math doesn't add up
Reads from PAYCHECK_HEADER (status = voided / amended) + PAYCHECK_LINE_ITEM (ytd_amount delta)
S8
Active only Watch Paycheck table
In-progress payroll gate
Detects whether a payroll run is currently in progress in QBDT (unsent direct deposit paychecks, open pay period). Hard gate — migration cannot begin until this is resolved.
Why this screen exists
QBDT creates PAYCHECK_HEADER records for a payroll run before direct deposit files are sent to Intuit. If migration runs while an open payroll batch exists, the in-progress paychecks are captured with status = pending — not processed. The HistoricalPaycheck API will reject pending-status paychecks, causing the entire batch to fail mid-run with no recovery path noted in the PRD
The in-progress payroll's YTD figures are incomplete — the paycheck is counted but the money hasn't moved yet. Migrating YTD with a pending paycheck produces an over-counted YTD that causes the first QBO payroll run to over-withhold
Information captured
Detection: any PAYCHECK_HEADER with direct_deposit_flag = true and status = pending within the migration window
Hard block message: "A payroll for [pay period dates] is currently being processed in QuickBooks Desktop. Complete and send this payroll before migrating."
Re-scan option — customer completes payroll in QBDT, returns to this screen, and re-runs the check. Screen clears when no pending records are found
What breaks if skipped
Pending paycheck in payload → API rejection → partial migration with no recovery path → employees who had pending paychecks have no historical record in QBO at all
Over-counted YTD → first QBO payroll over-withholds federal and state income tax → employee short-pay
Reads from PAYCHECK_HEADER (status = pending, direct_deposit_flag)
S9
Active + Inactive Pre-flight Paycheck table Tax table Pay item table
Pre-flight validation report — dry run before migration commits
Silent API dry-run against every confirmed mapping. Surfaces validation failures, math-error candidates (read-only overrides), and deduplication conflicts before any data moves. ProAdvisor resolves; migration does not start until this screen is clear.
Why this screen exists
PRD Risk 2: "Historical paychecks in DT might have mathematical inaccuracies that QBO's strict tax engine rejects." A dry run catches these before committing. Without a pre-flight pass, math-error rejections surface mid-migration — the customer has already confirmed their mappings and expects everything to complete. There is no designed recovery path in the PRD for mid-run failures
PRD Risk 1: deduplication — the pre-flight checks that each paycheck record, if migrated as a HistoricalPaycheck object, will correctly suppress its corresponding JOURNAL_ENTRY bank transaction. Any paycheck that cannot be deduplicated is flagged before it creates a doubled CoA entry
Pre-flight categories surfaced
Ready — passes all payload validations, will migrate as native paycheck object
Read-only override — math error detected; QBO will accept as historical fact but paycheck cannot be edited post-migration. ProAdvisor confirms per batch or bulk-accepts
Blocked — payload will be rejected by API; ProAdvisor must choose: skip / archive as PDF / fix mapping
Dedup conflict — paycheck has no matching JOURNAL_ENTRY to suppress; migrating will create a new CoA entry. ProAdvisor confirms this is expected or investigates
What breaks if skipped
Math-error paychecks fail mid-migration → partial success state with no recovery path → some employees have historical records, others don't → inconsistent Pay History
Dedup failure → doubled CoA entry → Balance Sheet shows double the payroll expense for that period → accountant review required, support escalation
Reads from PAYCHECK_HEADER + PAYCHECK_LINE_ITEM + PAYCHECK_TAX + JOURNAL_ENTRY (reconciled_flag)
S10
Active + Inactive Business Owner Paycheck table
Migration readiness summary & Business Owner authorisation
Single-screen non-technical summary for the Business Owner: paycheck counts, employee tiers, estimated duration, pre-flight outcomes. Three decisions, named authorisation, dated receipt. Migration does not start without this.
Why this screen exists
Migration authorisation is a business decision, not a payroll decision. The Business Owner must explicitly agree that the defined period is correct, that no payroll is in progress, and that the ProAdvisor is authorised to run the migration on their behalf. Without a named, dated authorisation, there is no audit record of who initiated the migration — critical for support escalation and compliance review
This screen is shown only to the Business Owner — it contains no payroll taxonomy. The summary uses employee names and date ranges only. The BO is not asked to understand payroll item mappings or API validation results
Three decisions captured
1. Period confirmation: "We will migrate paychecks from [date] to [date] for [X] employees. Is this correct?"
2. In-progress declaration: "No payroll is currently being processed in QuickBooks Desktop." (Checkbox — hard-gates on S8 result)
3. Named authorisation: "I, [full name], authorise [ProAdvisor name] to migrate payroll history for [company name] on [date]."
Confirmation receipt generated — emailed to both BO and PA, stored in QBO audit trail
What breaks if skipped
No audit record of who authorised migration → if a dispute arises post-migration, there is no evidence trail → support escalation becomes a he-said/she-said
BO discovers post-migration that the wrong period was migrated → no record of what was agreed → rollback conversation without a baseline
Reads from PAYCHECK_HEADER (count, date range) + EMP_MASTER (employee count, tiers)

S13
Inactive — Tier 3ArchivePaycheck tablePay item table
Tier 3 archive creation — inactive employee structured storage
Inactive employees whose EMP_MASTER data is insufficient for a live HistoricalPaycheck API post are stored as structured archive objects. Every PAYCHECK_LINE_ITEM field preserved as discrete queryable fields — not a PDF or string blob.
Why this screen exists
An inactive employee whose address or W-4 is missing cannot satisfy the QBO HistoricalPaycheck API's minimum payload. But their paycheck history is still needed for: W-2 correction eligibility (3 years), housing verification requests (PRD Section 6 north-star), labor audits, and garnishment compliance. The archive preserves every line item as a structured, queryable field — not a flat file — so the data remains reportable without a live QBO employee record
Archive fields preserved per paycheck
All PAYCHECK_HEADER fields: paycheck_date, period_start, period_end, gross_pay, net_pay, check_number, status
All PAYCHECK_LINE_ITEM fields: pay_item_name, amount, hours, rate, ytd_amount — per line item, not summed
All PAYCHECK_TAX fields: tax_type, jurisdiction, ee_amount, er_amount, ytd_ee, ytd_er
Migration metadata: migration_date, mapping_version_id, PA_name, archive_reason (insufficient EMP_MASTER data / prior-year scope)
Re-link flag: employee_id tagged as re-linkable if employee is rehired and a new QBO record is created
What breaks if skipped
Inactive employee paychecks silently convert to bank transactions (current state) — the exact PRD problem, unresolved for this employee segment
W-2 correction: if a W-2c is needed for an inactive employee and the paycheck detail is not preserved, the correction requires manual reconstruction from paper records
Reads from PAYCHECK_HEADER + PAYCHECK_LINE_ITEM + PAYCHECK_TAX → Writes to QBO Archive store

ActiveCritical — complianceTax tablePaycheck table
Per-batch YTD validation sign-off
After each employee batch, side-by-side QBDT vs QBO YTD diff per line item. PA signs off. BO co-signs if any delta exceeds the configurable threshold. AI explains every amber delta in plain English.
Why this screen exists
The QBO YTD accumulator (built from the migrated HistoricalPaycheck objects) must exactly match the QBDT YTD at the migration date. The first live QBO payroll run reads this accumulator to calculate year-to-date withholding and determine whether any FICA or other tax caps have been reached. An incorrect accumulator produces incorrect withholding from the first QBO paycheck
For rehired employees: the YTD sign-off must show two separate rows — one per employment period — confirming that the split was applied correctly and each period's YTD is independently accurate

Gross wages YTD (from PAYCHECK_LINE_ITEM.ytd_amount sum)
Federal income tax withheld YTD (PAYCHECK_TAX.ytd_ee where tax_type = FIT)
Social Security wages and tax YTD (FICA EE + ER)
Medicare wages and tax YTD
State income tax YTD per jurisdiction
Each deduction YTD (401k, health insurance, garnishment)
What breaks if skipped
YTD mismatch undetected → first QBO payroll run produces incorrect withholding → employee short-pay or over-pay → immediate support call and potential PRD escalation
Compares PAYCHECK_TAX.ytd fields (QBDT source) vs QBO YTD accumulator (target)

Check-to-paycheck intercept & recovery screen
Catches paychecks the native QBO migration tool converted to bank transactions. Re-matches using multi-field key. Re-posts as HistoricalPaycheck. Unmatched conversions surface with three resolution choices.
Why this screen exists
PRD identifies this as the root cause of the lighthouse issue: paychecks landing as bank transactions in the CoA. The native migration tool converts paychecks to checks by design. The intercept layer is the mechanism that overrides this behavior — but it depends on matching the converted check back to its PAYCHECK_HEADER source using a multi-field key. This screen shows the PA which paychecks were recovered, which were unmatched, and why
Match key fields used
paycheck_date matches JOURNAL_ENTRY.transaction_date
net_pay matches JOURNAL_ENTRY.credit_amount on bank account
employee_id matches via SSN cross-reference
pay_period_end matches PAYCHECK_HEADER.pay_period_end
